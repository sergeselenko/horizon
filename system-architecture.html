The Session Flow

From startup to knowledge capture — how every session runs

1

Startup Reads

Claude reads Notion pages via MCP before its first substantive response. Core projects have their own reading order defined in their instructions. Area-specific projects read: Global Instructions → Global Operator Profile → Project Instructions → Project Snapshot → Knowledge Digest → optional 6th page. Context budget: approximately 20-40K tokens.

~20-40K tokens of context loaded
2

Auto-Registration

After startup reads, Claude creates a lightweight Session Log entry with status "In Progress." This marks the session as active in the Session Tracker. Entry includes: chat name, project, area, date, timestamp. All other fields left blank for /closeout to fill.

Session Log entry created (In Progress)
3

Work

The substantive session. Claude follows Global Instructions behavioral rules throughout. Plan → Approve → Execute → Confirm cycle for all substantive work. DDV pattern for significant Notion operations. Thinking mode declarations and cognitive phase labeling for complex deliverables.

Substantive outputs produced
4

/closeout

End-of-session ritual. Claude reads the /closeout prompt page and follows it step by step. Recommends keep/delete. Creates or updates Session Log entry (fills in Summary, Key Outputs, Pending Items, Handoff Prompt). Runs Continuity Check: is the snapshot current? Is the Knowledge Digest current? Are engagement documents up to date? Registers the session on the Chat Map.

Session Log filled, Chat Map updated
5

/tasks

Scans the conversation for future tasks to capture to Notion. Run separately after /closeout. Each task links to a project — no orphaned tasks. Creates entries in the Tasks database.

Tasks captured to Notion
6

/capture

Harvests instruction improvements, decision records, and workflow documentation. The learning loop — flags from the session become candidates for Global Instructions updates. Presents each category with approval gates between them.

Slash Commands

8 commands available across all projects

/closeout
Prompt page in Notion▸
Close out a session — log, map, and continuity check
Recommends keep/delete for the chat. Creates or updates Session Log entry with Summary, Key Outputs, Pending Items, and Handoff Prompt. Runs Continuity Check: is the snapshot current? Is the Knowledge Digest current? Are engagement documents up to date? Registers the session on the Chat Map. Has mandatory approval stops — Claude must wait for confirmation at each gate.
/tasks
Prompt page in Notion▸
Scan conversation for tasks to capture to Notion
Scans the entire conversation for future tasks that should be tracked. Creates entries in the Tasks database with project links. Run separately after /closeout. Every task must link to a project — no orphaned tasks.
/capture
Prompt page in Notion▸
Knowledge capture — improvements, decisions, workflows
Harvests instruction improvements, decision records, and workflow documentation. Presents each category with approval gates between them. Searches for /flag markers from the session and evaluates each as a potential instruction improvement candidate. The learning loop that makes Global Instructions better over time.
/digest
Prompt page in Notion▸
Update a venture's Knowledge Digest with new domain intelligence
Updates a venture's Knowledge Digest with new domain intelligence — competitive data, pricing, people observations, strategic patterns. Standalone command — can run mid-session or after /closeout flags a gap. Presents plan before executing.
/emit
Prompt page in Notion▸
Mid-session cross-project state propagation
Scans conversation for decisions, actions, new information, cross-venture implications, and status changes. Writes to Global Operator Profile (cross-project broadcast) and Secretary Pad (Secretary awareness). Optionally updates Venture Posture Table if venture posture changed. Every future session reads these at startup — state changes propagate automatically.
/flag
Inline — no prompt page▸
Mid-session course correction bookmark
User types /flag — [description]. Claude acknowledges in one line and continues working. The flag is a bookmark for later extraction — not an invitation to debug. During /capture, flags are harvested and evaluated as instruction improvement candidates.
/scope
Inline — no prompt page▸
Session scope lock — prevent drift
User types /scope — [description]. Claude flags drift outside declared scope: "That's outside our declared scope. Want to add it, or save for a separate chat?" Can be expanded mid-session with natural language. Scope constrains what the session produces, not what it references.
/snapshot
MCP-native▸
Lightweight mid-session snapshot update
Quick save of current project state without full closeout. MCP-native — no prompt page needed. Updates the project snapshot page in Notion with current state. Use when you want to preserve progress mid-session without going through the full /closeout ritual.

The Three-Tier Architecture

How instructions reach Claude — from static to living to reference

1

Custom Instructions

Thin text pasted into each Claude Project's settings. Points Claude to the Notion pages it needs to read. Contains circuit breaker rules — the highest-damage failure mode protections that must always be active. Rarely changes. Approximately 3-5K tokens per project. This is the lightest layer: it exists to bootstrap the Notion connection, not to carry content.

2

Notion Living Layer

The source of truth. Global Instructions, project instruction pages, operator profile, snapshots, digests — all live here. Updated frequently. Read by Claude via MCP at startup. This is where anything that changes lives. The principle: update once in Notion, every project picks it up automatically. This tier contains the vast majority of the system's intelligence — behavioral rules, operational procedures, strategic context, and domain knowledge.

3

Project Files

Static reference files attached to Claude Projects. PDFs, research reports, markdown files. Used for content that rarely changes and provides deep reference context. Must stay aligned with Notion — if they conflict, Notion wins. Checked for staleness when instructions update. These are supplementary, not authoritative.

Hosting Infrastructure

Where artifacts live and how they're secured

Repository: github.com/sergeselenko/apps — public repo, GitHub Pages enabled
Live URL base: sergeselenko.github.io/apps/
Authentication: GitHub Personal Access Token embedded in git remote URL
Security: PIN gate (SHA-256, PIN 1930) — obscurity-level, not encryption. Fine for dashboards, not for PII.

Push Workflow

From Claude to live page

Primary — CoWork Project
"apps" project in Claude Desktop. Folder instruction triggers git add/commit/push on new files. Only runs while Mac is awake. Automatic deployment when files land in the folder.
Backup — Terminal Alias
dailypush alias in ~/.zshrc. One-word command. Manual fallback when CoWork isn't running.
cd ~/Documents/apps && git add -A && git commit -m "update" && git push

Artifact Registry & Design Consistency

Tracking and standards for all hosted artifacts

Artifact Registry
All hosted artifacts are cataloged in the Artifact Registry Notion database. Tracks: artifact name, filename, live URL, source project, description, dates, PIN gate status. Manually maintained after each push.
Design Consistency
All HTML artifacts follow the Selenko Artifact Design System (Notion page). Hub chats producing delegation prompts use the HTML Artifact Build Workflow and its handoff template. This ensures visual consistency across all hosted artifacts — same fonts, colors, components, and interactions.
