ArchitectActive▸
System design layer. Builds, refines, and evolves the entire Claude Projects ecosystem.
What happens here: Writing and refining Global Instructions, designing new project structures, evolving Notion architecture, planning automation (N8N/MCP), and onboarding new ventures when they graduate from Operator Advisory mode.

OperatorActive▸
Always-available operations partner. Five modes via auto-detection or slash commands.
Note ProcessingDaily OrchestrationReal-time AssistantBusiness AdvisoryVenture Mode
Note Processing (/process): Takes raw inputs — voice memos, screenshots, photos, pasted text, transcripts — classifies them by area (9 areas), routes them to the correct Notion location (tasks, projects, contacts, decisions), and produces a structured summary.
Daily Orchestration (/daily): Reads Tasks DB + Calendar + Global Operator Profile. Produces a prioritized Daily Outlook: what's due, what's urgent, what's important. Supports mid-day re-planning.
Real-time Assistant (/secretary): Thinking partner and context broker. Reads the Secretary Pad at startup to pick up cross-session state. Handles brain dumps, meeting prep, quick strategic questions, cross-venture thinking.
Business Advisory (/advisory): Structured intake for new ventures using the Intake Framework (Vision, Status, People, Revenue Model, Systems, Blockers, Next Actions). When ready, produces a handoff to the Architect.
Venture Mode (/activemenus, /selenko, etc.): Deep context loading for one venture — reads snapshot, Knowledge Digest, and Key Person Profiles on top of standard startup reads.
Mode detection: Auto-detects from the first message. Paste raw text → Note Processing. Ask about your day → Daily Orchestration. Brain dump → Real-time Assistant. Override anytime with a slash command.
Connections: Reads → Global Instructions, Global Operator Profile, Secretary Pad, venture snapshots/digests. Writes → Tasks DB, Secretary Pad, Session Log. Hands off to → Architect (system design), area projects (deep work).

Session Lifecycle
Every Claude chat follows the same lifecycle: startup reads load context from Notion, auto-registration marks the session as active, work happens with approval gates, and the session closes with /closeout → /tasks → /capture. Eight slash commands control different parts of this flow — each one is a text trigger that makes Claude read a procedure from Notion and follow it step by step with mandatory approval stops.
Session FlowEvery session progresses through these five stages, from the first message you send to the final /capture. The approval gate (Plan → Approve → Execute → Confirm) sits in the middle of the "Work" stage and prevents Claude from doing anything substantive without your explicit go-ahead.

Slash CommandsThese are not platform features — they're a text convention. You type /closeout (or any command below) in any chat, Claude recognizes the text, reads a step-by-step procedure from a Notion page, and follows it with mandatory approval stops. Each command does one specific job. They never auto-chain — you trigger each one separately.
/closeout▸
Closes out a session — recommends keep or delete, creates a Session Log entry, checks that snapshots and digests are current.
What it does: Recommends keep or delete for the chat. Creates/updates a Session Log database entry with summary, key outputs, pending items, handoff prompt, tags, and cross-venture connections. Runs a Continuity Check: is the project snapshot current? Is the Knowledge Digest current? Are engagement documents up to date?

Approval gates: Multiple mandatory stops — Claude presents the Session Log entry for approval before writing, and the keep/delete recommendation before acting.

When: End of any substantive session. Always run before leaving a chat you want to preserve.

Reads: /closeout prompt page. Writes: Session Log DB, Chat Map. Triggers: /tasks and /capture suggestions (never auto-chains).
/tasks▸
Scans the entire conversation for action items, to-dos, and commitments, then creates task entries in Notion.
What it does: Scans the conversation for tasks. Presents each as a proposed entry with Project, Area, Status, Context, and Due Date. Creates entries in Tasks DB after approval. Every task must link to a project.

When: After /closeout, before /capture.

Writes: Tasks DB.
/capture▸
The learning loop — harvests behavioral improvements, decision records, and workflow documentation from the session.
Three categories with approval gates between each:

Category 1 — Instruction Improvements: Scans for behavioral failures, /flag markers, and moments where Claude went off-track. Each becomes an Improvement Log entry with Evidence, Suggestion, Success Criterion, and Effectiveness tracking.

Category 2 — Decision Records: Identifies significant decisions that should be formally documented.

Category 3 — Workflow Documentation: Captures new processes that emerged and should be formalized.

Reads: /capture prompt page, searches for /flag markers. Writes: Improvement Log DB. Downstream: Approved improvements → Global Instructions (via Architect).
/emit▸
A cross-project broadcasting command. Scans the conversation for important decisions and writes them to shared surfaces so every future session picks them up.
How it works: Scans the current conversation for decisions, actions, new information, and status changes. Writes to two Notion pages:

Target 1 — Global Operator Profile: "Recent Cross-Venture Activity" section. Every project reads this at startup, so information propagates automatically to all future sessions.

Target 2 — Secretary Pad: Activity Log section. The Real-time Assistant reads this, maintaining cross-session awareness.

Optional: Updates the Venture Posture Table if a venture's posture changed (going active, pausing, restructuring).

When: Mid-session, whenever something happens that other projects should know about.

Reads: Current conversation. Writes: Global Operator Profile + Secretary Pad. Consumed by: All future sessions across all projects.
/digest▸
Updates a venture's Knowledge Digest — the domain knowledge page that captures competitive data, pricing, people observations, and strategic patterns.
When: After a session that surfaced significant domain knowledge. Can run mid-session or after /closeout flags a gap.

Reads: /digest prompt page + current Knowledge Digest. Writes: Venture's Knowledge Digest page. Read by: All future sessions in that venture (startup read #5).
/snapshot▸
A lightweight, mid-session save. Updates the Project Snapshot page — the two-part document (Latest Session + Comprehensive State) that captures where a venture stands.
When: You want to save progress without the full /closeout sequence. Useful when switching contexts.

Scoped to: Project Snapshot page only. Does not touch Session Log or Chat Map.
/flag▸
A mid-session bookmark for when Claude goes off-track. Type it, Claude acknowledges in one line and continues working. The flag gets harvested later by /capture.
Example: /flag — ran 5 searches before reading the obvious page

During /capture, Category 1 searches for all /flag markers and evaluates each as a potential instruction improvement.

Note: Unlike other commands, /flag runs inline — no Notion prompt page to read.
/scope▸
Declares a boundary for what this session will cover. After declaration, if the conversation drifts outside scope, Claude flags it and asks whether to expand or save for another chat.
Expand scope: Say "add [X] to scope" at any time. No formal syntax needed.

Optional: Sessions without /scope work normally. Scope constrains what the session produces, not what it references — Claude can still read any Notion page for context.
System Infrastructure
Everything lives in Notion under the Claude System Hub. This tab shows every page, database, index, and template that powers the system — organized by type and by when they get loaded. "Startup" pages are read automatically at the beginning of every session via MCP. "On-demand" pages are consulted when a specific topic or person comes up. Click any card to see what it contains, its Notion page ID, and which parts of the system read from or write to it.
Three-Tier ArchitectureThe system uses three layers of instructions. Tier 1 is pasted into Claude's project settings and rarely changes. Tier 2 lives in Notion and gets updated constantly — this is the living heart of the system. Tier 3 is static reference files loaded into the context window. If Tier 2 and Tier 3 ever conflict, Tier 2 wins.
Tier 1
Custom Instructions
Thin pointers in Claude Project settings. Circuit breakers + Notion URLs. ~3-5K tokens. Rarely changes.
Tier 2
Notion Living Layer
Instructions, snapshots, digests, profiles. Updated frequently. Read via MCP at startup. This is where anything that changes lives.
Tier 3
Project Files
Static reference. PDFs, research docs. Loaded into context window. Supplementary — Notion overrides conflicts.

Instruction Improvement Loop▸
The system gets smarter every session: /flag → /capture → Improvement Log (with Success Criterion) → Architect reviews → Global Instructions update → all future sessions enforce the rule → Effectiveness tracked.
Completeness-First Advisory▸
When Claude is the primary knowledge source on a complex topic, it generates a complete topic map, audits gaps proactively, and applies the "Unsophisticated Client Test." Two-tier: trigger in Global Instructions + full reference page.
Expert Analysis Review▸
Mandatory quality gate for complex deliverables. Claude produces the work, then produces a handoff prompt for a dedicated review chat that cross-references against source material independently.
Standardized Snapshot Format▸
Two parts: (1) Latest Session — what just happened. (2) Comprehensive State — the full cumulative picture. Information is never dropped between snapshots. Archive-before-replace workflow.
Thinking Mode Declarations▸
Claude declares its cognitive mode (Evaluation, Solution Design, Synthesis, Critique, Brainstorm) at the start of responses + confirms in footer. Makes reasoning visible so you can redirect.
Cognitive Phase Labeling▸
Four phases for complex deliverables: Analyze → Design → Build → Verify. The approval gate sits between Design and Build. Prevents premature execution.
Context Window Footer▸
Every response ends with: ~XK tokens used | what's consuming it | compaction risk + thinking mode. Always-on ambient awareness of context health.
