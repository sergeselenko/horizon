Hub-and-Spoke Chat Pattern

Two chat types that work together — strategy and production

Hub Chats
The continuity thread within a project. See the big picture, make strategic decisions, design solutions, delegate production work. Don't build things except quick MCP actions (≤3 pages). Version number tracks evolution of thinking.
Examples: Architect v7 ActiveMenus v4
Build Chats
Receive delegation prompts from hub chats. Produce specific deliverables. Don't make strategic decisions or expand scope. Read Global Instructions, auto-register in Session Log, surface Session Log URL back to user.
Examples: System Architecture Dashboard v2 Portfolio Map v1
How They Connect
Build chats read Global Instructions at startup, auto-register in Session Log ("In Progress"), and surface their Session Log URL. The user brings that URL to the hub chat, creating the link. Hub chats track delegated builds during /closeout.

The Session Flow

From startup to knowledge capture — both hub chats and build chats follow this flow

1

Startup Reads

Claude reads Notion pages via MCP before its first substantive response. Core projects have their own reading order. Area-specific projects read: Global Instructions → Global Operator Profile → Project Instructions → Project Snapshot → Knowledge Digest → optional 6th page. Build chats read Global Instructions + any Notion pages specified in their delegation prompt. Context budget: ~20-40K tokens.

~20-40K tokens of context loaded
2

Auto-Registration

After startup reads, Claude creates a lightweight Session Log entry with status "In Progress." Marks the session as active in the Session Tracker. Entry includes: chat name, project, area, date, timestamp. All other fields left blank for /closeout to fill.

Session Log entry created (In Progress)
3

Session Log LinkageBuild Chats

Build chats surface their Session Log URL in their first response. The user brings this URL back to the hub chat, creating a traceable link between the delegation and its output. Hub chats reference these URLs during /closeout when tracking delegated work.

Session Log URL surfaced → user brings to hub chat
4

Work

The substantive session. Claude follows Global Instructions behavioral rules throughout. Plan → Approve → Execute → Confirm cycle for all substantive work. DDV pattern for significant Notion operations. Thinking mode declarations and cognitive phase labeling for complex deliverables. Build chats follow their delegation prompt's phased build process.

Substantive outputs produced
5

/closeout

End-of-session ritual. Claude reads the /closeout prompt page step by step. Recommends keep/delete. Creates or updates Session Log entry (Summary, Key Outputs, Pending Items, Handoff Prompt). Runs Continuity Check. Registers on Chat Map. Hub chats also track delegated builds.

Session Log filled, Chat Map updated
6

/tasks

Scans the conversation for future tasks to capture to Notion. Run separately after /closeout. Each task links to a project — no orphaned tasks.

Tasks captured to Notion
7

/capture

Harvests instruction improvements, decision records, and workflow documentation. The learning loop — flags from the session become candidates for Global Instructions updates. Presents each category with approval gates.

Quick save of current project state without full closeout. MCP-native — no prompt page.

Communication Output Types

Three output types for emails and messages

Chat Draft
Claude displays the email text in a code block inside the chat. Default first step for any email that needs iteration. Fast to produce, easy to review and refine. Copy-paste into Gmail when ready.
"Show me a chat draft"
Message Compose
Branded UI widget rendered inline. Shows the email with formatting, subject line, and "Open in Mail" button. Can present multiple strategic variants side by side.
"Message compose" · "put it in the widget"
Gmail Draft
Claude creates a draft directly in your Gmail Drafts folder via MCP. Can be created as a reply to an existing thread to preserve threading.
"Create a Gmail draft"
Recommended: 1. Chat Draft→2. Message Compose(optional)→3. Gmail Draft

The Three-Tier Architecture

How instructions reach Claude — from static to living to reference

1

Custom Instructions

Thin text pasted into each Claude Project's settings. Points Claude to the Notion pages it needs to read. Contains circuit breaker rules. Rarely changes. ~3-5K tokens per project.

2

Notion Living Layer

The source of truth. Global Instructions, project instruction pages, operator profile, snapshots, digests — all live here. Updated frequently. Read by Claude via MCP at startup. Update once in Notion, every project picks it up automatically. This tier contains the vast majority of the system's intelligence.

3

Project Files

Static reference files attached to Claude Projects. PDFs, research reports, markdown files. Must stay aligned with Notion — if they conflict, Notion wins. Supplementary, not authoritative.

Global Instructions — Tier 2 Architecture

On-demand rules loaded by trigger conditions — reduces always-loaded GI from ~15-18K to ~5-7K tokens
